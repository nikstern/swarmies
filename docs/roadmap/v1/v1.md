# v1

## Goal

Build a minimal agent orchestration loop that:

- pulls ready work from Beads
- selects an agent that can perform the work
- dispatches the work to that agent
- tracks task ownership and completion in Beads
- establishes the baseline A2A and ADK integration needed for future expansion

The v1 system should prove the end-to-end workflow before adding richer planning,
UI, or advanced agent selection.

v1 should also leave the system on the correct architectural path, which means
the initial implementation must include basic A2A and ADK setup rather than
treating them as post-v1 additions.

The concrete runtime contract for this roadmap is defined in
[v1 Runtime Architecture](../../architecture/v1-runtime.md).

## Core Flow

1. A Beads-backed client asks for ready work.
2. The dispatcher inspects the available task and its metadata.
3. The dispatcher selects an agent profile for that task.
4. The dispatcher sends the task to the selected agent over an A2A boundary.
5. The agent claims the Beads task before starting work.
6. The agent performs the work and returns a result to the dispatcher.
7. The dispatcher records the outcome and closes the task when complete.

## v1 Components

### 1. Beads Client

Responsible for all interaction with `bd`.

- query ready work
- show task details
- claim tasks atomically
- close completed tasks
- surface task metadata to the dispatcher

This can start as a thin Go wrapper around the `bd` CLI.

### 2. Dispatcher

Responsible for the control loop.

- poll or request ready work from the Beads client
- normalize task input into an internal work item
- select an agent profile
- dispatch the task
- receive results
- decide whether to close the task, retry, or leave it open

The dispatcher is the system entrypoint in v1.

### 3. Agent Interface

Responsible for executing assigned work.

- accept a task payload
- claim the task before execution
- report status and final result
- return structured success or failure information

For v1, this interface can be minimal and strongly typed in Go even if the
transport evolves later.

### 4. A2A Layer

Responsible for the transport boundary between dispatcher and agents.

- use the standard A2A agent card and `message/send` boundary
- handle request and response exchange
- isolate agent execution from the dispatcher process
- provide the minimum viable communication contract for v1

For v1, Swarmies should keep a thin local adapter, but the external contract
should remain A2A-native rather than a custom Swarmies protocol.

### 5. ADK Integration

Responsible for agent construction and execution lifecycle.

- define the baseline agent runtime shape
- provide a consistent way to register agent profiles
- encapsulate tool access, prompts, and execution context
- keep future agent capability growth compatible with the initial runtime

For v1, ADK does not need advanced planning. It only needs to provide the
minimal structure for an A2A-exposed agent with a loader and session service.

### 6. Agent Profiles

Responsible for matching task types to execution behavior.

Initial profiles may be simple:

- `generalist`
- `research`
- `coding`

Selection can begin with rules based on task labels, type, or description.

## Delivery Phases

### Phase 1: Single-Agent Happy Path

Build the smallest possible working loop.

- implement Beads CLI wrapper in Go
- set up the minimal ADK-based agent runtime and session service
- establish a minimal A2A `message/send` task-oriented contract
- fetch one ready task
- dispatch to one hard-coded agent
- claim the task
- simulate or perform work
- return a result
- close the task on success

Success for this phase means one task can move from ready to closed through the
system without manual coordination outside Beads.

### Phase 2: Typed Dispatch

Introduce basic routing logic.

- define an internal task model
- map Beads task data into that model
- select among multiple agent profiles
- capture structured execution status
- handle failures without corrupting task ownership

Success for this phase means the dispatcher can route different categories of
work to different agents with predictable behavior.

### Phase 3: A2A Boundary

Harden the initial A2A and ADK setup into a stable execution boundary.

- expand the initial request and response payloads
- harden A2A transport for dispatch
- support remote or independent agent execution
- improve ADK-backed agent registration and runtime configuration
- preserve Beads task lifecycle guarantees

Success for this phase means agent execution is decoupled from the dispatcher.

## Initial Constraints

- use Go as the primary implementation language
- use Beads as the source of truth for task state
- include A2A and ADK from the start, even if the first integration is minimal
- keep v1 synchronous where possible
- optimize for clarity and correctness over throughput
- prefer one reliable execution path before adding concurrency
- prefer A2A-native external types over custom transport envelopes
- keep ADK runtime concerns behind the agent boundary

## Open Questions

- what metadata in Beads is required for routing
- what result schema is sufficient for closing or retrying tasks
- how should partial progress be recorded in Beads
- what failure modes require requeue instead of close
- when should v1 move from in-memory ADK sessions to persistent storage
- whether v1 should accept direct A2A message responses or require task-shaped
  responses only

## Out of Scope for v1

- UI work with A2UI
- advanced planning or hierarchical decomposition
- probabilistic task selection such as SayCan
- multi-agent collaboration on a single task
- autonomous long-running background orchestration

## Definition of Done

v1 is complete when:

- a ready Beads task can be discovered by the system
- the dispatcher can select an agent for that task
- the dispatcher and agent communicate through an initial A2A `message/send`
  contract
- the selected agent runs inside a minimal ADK-backed runtime with a session
  service
- the selected agent can claim and execute the task
- the result is returned in a structured form
- successful work closes the task in Beads
- failure leaves enough state for safe inspection or retry
